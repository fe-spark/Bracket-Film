# Bracket-Film 常见问题（FAQ）

本文整理项目开发与运维中最常见的问题，优先覆盖 `server-v2` 与 `web` 的近期变更。

## 1. 切换主采集站会发生什么？

触发条件（任一满足即触发“主站变更”）：
- 附属站升级为主站（`slave -> master`）
- 主站还是主站，但主站 `URI` 改了

系统动作：
- 清空主站相关核心数据：`search_infos`、`movie_detail_infos`、`category_persistents`、`virtual_picture_queues`
- 清理影片相关缓存（`ClearCache()`）
- **保留** `movie_playlists`（附属站播放列表），用于主站切换后尽量复用播放源

目的：
- 避免“新主站 mid 与旧主站 mid 不一致”导致的数据错绑。

---

## 2. 定时采集逻辑是什么？

采集顺序速览：
- 自动采集（`Model=0`）：**主站阶段 → 附属站阶段**
- 一键批量（`Model=1` / 管理页一键采集）：**主站阶段 → 附属站阶段**
- 单站手动采集：按你点选的站点直接执行（主站/附属站）
- 多主站场景：主站阶段内可并行，但附属站阶段一定在主站阶段整体完成后开始

定时任务由 `FilmCollectTask.Model` 区分：
- `0`：自动采集所有已启用站点（`AutoCollect(time)`）
- `1`：采集指定站点（`BatchCollect(time, ids...)`）
- `2`：失败记录恢复（`FullRecoverSpider()`）
- `3`：孤儿播放列表清理（`CleanOrphanPlaylists()`）

`Model=0`（自动采集）当前为**严格两阶段**：
1. 先采集所有已启用主站（阶段结束后等待完成）
2. 再判断主站数据是否就绪（`HasMasterData()`）
3. 主站数据就绪后才开始采集已启用附属站

并行策略：
- **多个主站在主站阶段内可以并行执行**（受并发上限控制）；
- 但附属站阶段一定在主站阶段整体完成后才会开始。

这意味着：
- 主站被禁用/删除/采集失败导致主数据未就绪时，附属站阶段会被跳过；
- 下一次定时任务会基于最新站点配置重新评估并执行。

批量采集（`BatchCollect`）与自动采集保持同样顺序约束：
- 若选择了主站与附属站，会先跑所选主站，再判断主数据是否就绪，最后才跑所选附属站；
- 若未选择主站且当前无启用主站/主数据未就绪，附属站阶段会被跳过。

默认初始化会创建（默认是关闭状态，需要手动启用）：
- 自动采集：`0 */20 * * * ?`（每 20 分钟）
- 失败恢复：`0 0 4 * * 0`（每周日 4 点）
- 孤儿清理：`0 0 0 * * *`（每天 0 点）

补充：
- 执行孤儿清理前会检测“主站是否正在采集”，若主站任务运行中会跳过本次清理，避免误删。

## 3. 什么是“孤儿数据”，又是如何判断的？

定义：
- `movie_playlists` 中存在，但在当前主站影片集合中找不到对应 `movie_key` 的播放列表记录。

判断流程：
1. 从 `search_infos` 取主站影片名。
2. 生成有效 key 集合：`hash(影片名)`，以及去掉“第一季”后缀后的变体 key。
3. 取 `movie_playlists` 的全部 `movie_key`。
4. 不在有效集合里的 key 判定为孤儿并删除。

保护机制：
- 若 `search_infos` 为空，直接跳过清理（避免主站清空阶段误删全部播放列表）。

## 4. 失败记录重试后会不会一直叠加？

**现状（已优化）**
- 同一待处理失败项（`origin_id + collect_type + page_number + hour + status=1`）不再重复插入。
- 当同一项再次失败时，系统会更新原记录并累加 `retry_count`。

**结果**
- 列表不会被重复失败刷爆。
- 同一条记录可持续反映“最新失败原因 + 累计重试次数”。

---

## 5. 为什么前端失败记录页显示了“重试次数”？

后端模型新增了字段：
- `retry_count int not null default 1`

前端管理页已增加“重试次数”列，用于快速识别频繁失败项。

---

## 6. 老库已经上线，新增字段怎么处理？

如果你不是全新库，通常不会自动补齐该字段（取决于初始化流程是否触发迁移）。

建议手工执行：

```sql
ALTER TABLE failure_records
ADD COLUMN retry_count INT NOT NULL DEFAULT 1 COMMENT '重试累计次数';
```

可选优化（推荐）：

```sql
CREATE INDEX idx_failure_pending_key
ON failure_records (origin_id, collect_type, page_number, hour, status);
```

## 7. Popconfirm 背景透明怎么修？

**不要在页面里用 `:global` 覆盖。**

建议做法：
- 统一通过 `ConfigProvider` 的主题 token 管理浮层样式。
- 如果出现透明问题，优先排查是否手动把 `Popover` 的背景 token 指到了半透明变量。

---

## 8. `/api/provide/config` 里采集站入口怎么理解？

- 聚合入口：走本地数据库聚合结果。
- 采集站入口：带 `source` 参数时可直连对应采集站（按当前实现）。

建议：
- 日常用户默认走聚合入口。
- 排障、比对、应急时使用带 `source` 的入口。

---

## 9. 为什么“表存在但迁移没生效”？

因为初始化逻辑可能按“是否存在 `users` 表”来决定是否执行建表/迁移。老库常见情况是：
- 用户表已存在；
- 新字段迁移不会自动触发。

最佳实践：
- 为新增字段维护独立 SQL migration（手工或脚本化）。
- 不依赖“首次启动初始化”覆盖生产迁移。

---

## 10. 推荐排障顺序

1. 看管理页失败记录：是否集中在某站点/某页码。
2. 看 `retry_count`：高次数通常是源站稳定性问题。
3. 手动触发单条重试验证。
4. 必要时切换 `source` 直连对比聚合结果。
5. 检查 Redis 缓存是否命中旧数据。
