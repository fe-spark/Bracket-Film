# Bracket-Film 常见问题（FAQ）

本文档聚焦采集链路的 4 个核心问题

## 1. 采集站为什么需要设置主站才能工作？

说明：

- 主站负责写入核心影片数据（如 `search_infos`、`movie_detail_infos`、分类等），这些数据是前台列表、详情页、筛选能力的基础。
- 附属站主要提供播放列表（`movie_playlists`），它本身不承担完整影片元数据建设。
- 前端播放聚合逻辑会先拿主站详情，再拼接附属站播放源；没有主站数据时，附属站播放列表很难被稳定消费。

角色：

- 主站：影片主数据源
- 附属站：播放源补充层

---

## 2. 采集站和附属站的关系是什么？

定义：

- **主站（MasterCollect）**：负责“主数据”采集（影片基础信息与详情）。
- **附属站（SlaveCollect）**：负责“播放列表”采集（多线路、多站点补充）。

协作方式：

- 主站提供可检索、可展示的影片主干。
- 附属站为同一影片补充更多可播线路。
- 管理上，主站发生切换或主站 URI 变化时，会触发主站数据清理并重建；附属站播放列表会尽量保留以便复用。

---

## 3. 附属站是如何与主站数据建立关系的？

机制：通过 `movie_key` 进行哈希键关联。

写入时（附属站采集）：

- 以 `source_id + movie_key` 唯一键写入 `movie_playlists`。
- `movie_key` 会使用影片名哈希（`hash(name)`）；若有 `dbid`，也会额外写入 `hash(dbid)`。

读取聚合时（详情页拼源）：

- 系统会基于主站影片信息构造候选 key（如 `hash(dbid)`、`hash(name)`、去掉“第一季”后的名称哈希、部分副标题拆分哈希）。
- 逐个附属站查找匹配 `movie_key`，命中后拼到播放源列表中。

关联特性：

- 可兼容不同采集源命名差异，并在主站迁移时复用部分播放列表。
- 极少数重名或别名差异较大的影片可能出现匹配偏差，需人工排查。

---

## 4. 定时任务中的孤儿清理任务是什么？运行逻辑又是什么？

孤儿数据定义：

- `movie_playlists` 里存在，但无法在当前主站影片集合中找到对应 `movie_key` 的播放列表记录。

运行逻辑：

1. 读取主站影片名集合（来自 `search_infos`）。
2. 生成有效 key 集（`hash(name)` + 去掉“第一季”后缀的变体）。
3. 扫描 `movie_playlists` 中全部 `movie_key`。
4. 不在有效集内的 key 视为孤儿并删除。

保护机制：

- 若 `search_infos` 为空，直接跳过清理（防止主站重建期间误删全部播放列表）。
- 执行清理前会检查是否有主站正在采集；若主站采集中，本轮清理跳过，避免竞争条件下误删。

默认 cron：

- 孤儿清理任务默认表达式为 `0 0 0 * * *`（每天 0 点），初始化后默认关闭，需手动启用。

---

## 5. 主采集切换为附属站时，数据会发生什么变化？

- **数据不清理**：降级操作本身不会触发 `MasterFilmZero`（主数据清空）。原有采集的搜索索引（`search_infos`）、影片详情（`movie_detail_infos`）和分类数据将完整保留在数据库中。
- **采集逻辑变更**：降级后，该站点不再作为影片主元数据的来源，新采集的数据将全部转为播放列表（PlayList）存储在此站名下，并通过名称哈希（`movie_key`）与现有主数据进行动态关联。
- **状态影响**：如果这是系统中唯一的主站，降级后将失去元数据更新能力。前台仍可展示旧数据，但无法自动跟新集数或修复无效的详情信息。

---

## 6. 附属站切换为主采集站（等级提升）会如何？

- **即时清空元数据**：系统检测到提升为 `Master` 时，会执行 `MasterFilmZero` 清空所有全局元数据。
- **自动清理旧播放列表**：新版本中，提升为主站时系统会自动清理该站点在 `movie_playlist` 表中的旧有记录，防止其作为主站运行时产生冗余的“影子”数据。
- **保留其他源映射**：`movie_source_mappings` 会被保留，新主站采集重建详情后，播放源会通过名称或豆瓣 ID 哈希自动关联。
- **任务状态锁**：若当前有采集任务运行，系统将禁止切换等级，彻底杜绝“边删边写”的竞态问题。

---

## 7. 切换主采集站的最合理（推荐）操作流程是什么？

1. **停止任务**：确保所有活跃采集与定时任务已停止。
2. **执行切换**：更新站点等级（Grade）或采集地址（URI）。
3. **主站全量**：对新主站执行一次详情采集，重建元数据骨架。
4. **附属同步**：骨架就绪后再开启其他附属站的同步，实现源挂载。
5. **清理冗余**：必要时手动运行一次“孤儿清理任务”。

---

## 8. 技术视角：到底什么才算一次“主站切换”？

在系统代码逻辑中，仅以下两种行为被定义为“切换”并触发数据重置：
- **等级提升**：站点等级从 `Slave` 提升为 `Master`。
- **地址变更**：主站点的采集链接主机地址（URI）发生了修改。
- *注：修改站点备注名、采集间隔、同步图片等行为不会触发数据重置。*

---

## 9. 如果系统中已有一个主站，再将另一附属站提升为主站会怎样？

**系统已建立自动化强制机制，该操作是安全且受控的：**

1.  **单主站互斥**：当设置新主站时，系统会**自动将现有旧主站降级为附属站**，在数据库级别保证全局永远只有一个 Master。
2.  **数据重置与清理**：系统会执行 `MasterFilmZero` 清空旧主站的检索、详情和分类数据。同时，会**自动物理删除新主站旧有的播放列表记录**（防止其作为主站运行时产生冗余数据）。
3.  **内容级去重**：新采集的主站数据通过 `ContentKey` (豆瓣 ID 或片名哈希) 进行冲突检测。即使 ID 不同，系统也会通过指纹识别现有影片，合并元数据而非重复创建。
4.  **全局 ID 归约**：所有来源的数据通过 `MovieSourceMapping` 统一映射到全局 ID。主站切换后，只要内容指纹匹配，原本的播放历史和书签依然有效。
5.  **运行锁保护**：如果当前有任何采集任务正在进行，系统将**强制拦截（报错）等级切换请求**，确保数据重置不会在并发环境下导致损坏。
